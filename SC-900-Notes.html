Defense in Depth

00:07:39
Core Concept

Multiple layers of security so that if one layer fails, another layer prevents further access. No single point of failure.

The 7 Layers (Outside → Inside)

  1. Physical security — Protect datacenter hardware
  2. Identity & access — Control who can access what
  3. Perimeter — DDoS protection, firewalls
  4. Network — Limit communication between resources, deny by default
  5. Compute — Secure VMs, close ports, patch OS
  6. Application — Ensure apps are secure, store secrets properly
  7. Data — Encrypt data at rest and in transit

CIA Triad

00:09:51
Confidentiality

Protect data from unauthorized access. Encryption, access controls, need-to-know principle.

Integrity

Data has not been tampered with. Hashing, digital signatures, version control.

Availability

Systems are accessible when needed. Redundancy, disaster recovery, DDoS protection.

Zero Trust

00:11:54
Core Principle

"Never trust, always verify." Assume breach. Every request must be authenticated, authorized, and continuously validated regardless of where it originates.

Three Guiding Principles

  • Verify explicitly — Always authenticate and authorize based on all available data points (identity, location, device health, service/workload, data classification, anomalies)
  • Use least privilege access — Limit user access with Just-In-Time (JIT), Just-Enough-Access (JEA), risk-based adaptive policies
  • Assume breach — Minimize blast radius, segment access, encrypt end-to-end, use analytics to detect threats

Six Pillars of Zero Trust

Identities Devices Applications Data Infrastructure Networks

Each pillar is verified with signals → policy decision → enforcement. Microsoft Entra ID is the control plane for identity in Zero Trust.

⚡
Exam Tip: Zero Trust is the opposite of the old "castle and moat" (perimeter) model where once inside the network you're trusted.

Encryption Basics

00:16:48

🔄 Symmetric Encryption

  • Same key for encryption and decryption
  • Fast, efficient for large data
  • Key distribution is a challenge
  • Examples: AES, 3DES
  • Used for: data at rest, bulk data

🔀 Asymmetric Encryption

  • Public key encrypts, private key decrypts
  • Slower than symmetric
  • Solves key distribution problem
  • Examples: RSA, ECC
  • Used for: TLS handshakes, digital signatures

Data States & Encryption

StateDescriptionProtection Method
At restStored on disk, database, storageAES-256 encryption (Storage Service Encryption, TDE)
In transitMoving between systemsTLS 1.2/1.3, HTTPS, VPN
In useBeing processed in memoryConfidential computing, secure enclaves

Digital Signatures & Certificates

  • Digital signature: sender hashes data → encrypts hash with private key → receiver decrypts with public key to verify
  • Certificate Authority (CA): trusted third party that issues digital certificates
  • PKI (Public Key Infrastructure): the system for managing keys and certificates
  • HTTPS uses TLS which uses asymmetric encryption to exchange a symmetric session key

Hashing

00:22:53
Core Concept

A one-way function that converts any input into a fixed-length string. Cannot be reversed. Same input always produces same output.

Key Properties

  • One-way / irreversible — cannot derive original from hash
  • Deterministic — same input = same hash, always
  • Avalanche effect — tiny input change → completely different hash
  • Collision resistant — two different inputs shouldn't produce same hash

Common Algorithms

  • MD5 — 128-bit (deprecated, insecure)
  • SHA-1 — 160-bit (deprecated)
  • SHA-256 — 256-bit ✅ Recommended
  • SHA-512 — 512-bit ✅ Stronger

Use Cases

  • Password storage (never store plaintext)
  • File integrity verification
  • Digital signatures
  • Blockchain / data chains
Salting

Adding a random value (salt) to a password before hashing prevents rainbow table attacks. Each user gets a unique salt stored alongside the hash.

Identity Concepts

00:27:09
Identity as the New Perimeter

In a cloud-first world, identity is the primary security boundary. Microsoft Entra ID (formerly Azure Active Directory) is the identity control plane.

4 Pillars of Identity

PillarDescription
AdministrationCreation and management of identity (users, groups, devices)
AuthenticationThe "who are you?" challenge — verify the claimed identity
AuthorizationThe "what can you do?" check — what resources can you access
AuditingTrack what happened, by whom, when, and from where

Microsoft Entra ID Overview

  • Cloud-based Identity and Access Management (IAM) service
  • Used for signing into Microsoft 365, Azure, and thousands of SaaS apps
  • A tenant = a dedicated instance of Entra ID for an organization
  • Supports external identities (B2B, B2C)
  • Free tier available; P1/P2 tiers unlock advanced features
🔐

Authentication Methods

00:30:12

Authentication Factors

Something you know (password, PIN) Something you have (phone, smart card) Something you are (biometrics)

Passwordless Authentication Options

  • Windows Hello for Business — PIN or biometric tied to device
  • FIDO2 Security Keys — Hardware key (e.g. YubiKey) — most phishing resistant
  • Microsoft Authenticator App — Push notification or number matching
  • Passkeys — FIDO2 standard stored on device or in cloud
⚡
Exam Tip: FIDO2 keys are the most phishing-resistant authentication method. Passwordless is more secure than passwords because it removes the password as an attack vector.

Active Directory Domain Services (AD DS)

00:34:16
On-Premises Identity

Traditional on-premises directory service. Uses Kerberos/NTLM for authentication. Organises objects into domains, trees, and forests.

AD DS (On-Prem)

  • Kerberos authentication
  • Group Policy Objects (GPO)
  • OU structure (Organizational Units)
  • Domain Controllers
  • Trusts between domains

Microsoft Entra ID (Cloud)

  • SAML/OpenID Connect/OAuth2
  • Conditional Access Policies
  • Flat structure (no OUs/GPOs)
  • No Domain Controllers
  • Global service (worldwide)

Microsoft Entra Connect

Synchronises on-premises AD DS identities to Entra ID. Enables hybrid identity. Supports Password Hash Sync (PHS), Pass-through Authentication (PTA), and Federation.

🤝

Federation

00:36:17
Core Concept

Establishes trust between separate identity systems so users can authenticate once and access resources across systems (cross-organization SSO).

Key Federation Concepts

  • Identity Provider (IdP) — The system that authenticates the user (e.g. Entra ID)
  • Service Provider (SP) / Relying Party — The application that trusts the IdP
  • Claims — Assertions about the user (name, email, group membership)
  • Tokens — Encoded, signed claims passed from IdP to SP (SAML tokens, JWT)
  • Single Sign-On (SSO) — Log in once, access many apps without re-authenticating
SAML 2.0

XML-based standard for exchanging authentication/authorization data. Common for enterprise apps.

OpenID Connect (OIDC)

Modern identity layer built on OAuth 2.0. Uses JWT tokens. Common for modern web/mobile apps.

Multi-Factor Authentication (MFA)

00:45:12
Core Concept

Requiring 2 or more verification factors. Even if one factor is compromised, the attacker still can't authenticate.

MFA Methods (Microsoft)

  • Microsoft Authenticator app — Push notification with number matching
  • SMS / Voice call — Least secure MFA (SIM swapping risk)
  • OATH TOTP tokens — Time-based one-time passwords (e.g. Google Authenticator)
  • FIDO2 keys — Most secure, phishing resistant
  • Certificate-based auth — Smart card or software certificate

Conditional Access (CA)

The if/then policy engine of Entra ID. Signals → Decisions → Enforcement.

  • Signals: User/group, IP location, device platform, device compliance, application, real-time risk
  • Access decisions: Block access, Grant access, Require MFA, Require compliant device, Require terms of use
  • Requires Entra ID P1 or P2 licence
⚡
Exam Tip: Security Defaults (free tier) enable basic MFA for all users. Conditional Access (P1/P2) gives granular, policy-based control.

Authorization — RBAC & Access Management

00:51:19
Role-Based Access Control (RBAC)

Grant access based on roles rather than to individuals. WHO (principal) + WHAT (role definition) + WHERE (scope).

Azure RBAC Structure

  • Security principal: User, Group, Service Principal, Managed Identity
  • Role definition: Collection of permissions (Actions, NotActions, DataActions)
  • Scope: Management group → Subscription → Resource group → Resource
  • Roles are additive and inherited — lower scope inherits parent scope roles

Key Built-in Roles

RoleAccess Level
OwnerFull control + assign access to others
ContributorCreate/manage all resources, cannot assign roles
ReaderView resources only
User Access AdministratorManage user access only

Privileged Identity Management (PIM)

  • Provides Just-In-Time (JIT) privileged access
  • Users activate roles when needed (for a limited time)
  • Requires justification and optional approval
  • Full audit log of privilege use
  • Requires Entra ID P2

DDoS Protection

01:06:03
DDoS = Distributed Denial of Service

Flooding a target with traffic from many sources to make it unavailable. Attacks availability (the A in CIA).

Azure DDoS Protection Tiers

TierDetails
Network Protection (Basic)Free, automatic, always on for Azure infrastructure. Protects all Azure services.
IP ProtectionPer-IP paid tier. Enhanced mitigation, rapid response support, cost protection guarantee.
Network Protection (Premium)Per-VNet pricing. Telemetry, alerts, reports. DDoS Rapid Response team access.

Attack Types

  • Volumetric — Flood bandwidth (DNS amplification, UDP floods)
  • Protocol — Exploit network protocol weaknesses (SYN floods)
  • Application layer (L7) — HTTP floods targeting app logic (Azure WAF handles this)

Azure Firewall

01:07:39
Managed Cloud Firewall

Fully stateful, cloud-native firewall-as-a-service with built-in high availability and unrestricted cloud scalability.

Key Features

  • Network and application traffic filtering
  • FQDN (Fully Qualified Domain Name) filtering — e.g. *.microsoft.com
  • Threat intelligence-based filtering — blocks known malicious IPs/domains
  • Outbound SNAT, inbound DNAT
  • TLS inspection (Premium SKU)
  • Centralised policy management via Azure Firewall Manager
Azure Firewall Standard

L3-L7 filtering, threat intelligence, FQDN filtering. Good for most workloads.

Azure Firewall Premium

Adds TLS inspection, IDPS (Intrusion Detection and Prevention), URL filtering, web categories.

🛡️

Web Application Firewall (WAF)

01:08:33
Application Layer (L7) Protection

Protects web applications from common exploits like SQL injection, cross-site scripting (XSS), and OWASP Top 10 threats.

WAF Deployment Options

  • Azure Application Gateway — WAF for web apps in a region, supports SSL offload
  • Azure Front Door — Global WAF at Microsoft's edge/CDN layer
  • Azure CDN — WAF at edge (Akamai/Verizon profiles)
Detection Mode

Logs and monitors threats but does not block. Good for initial tuning.

Prevention Mode

Actively blocks malicious requests. Recommended for production.

Virtual Networks (VNET) & NSG

01:10:19

Azure Virtual Network (VNET)

  • Logical isolation of Azure network resources
  • Segmented into subnets
  • VNETs can be connected via peering
  • Private IP addresses within VNETs
  • No internet routing by default

Network Security Group (NSG)

  • Filter network traffic with security rules
  • Applied to subnets or NICs
  • Rules based on: source/dest IP, port, protocol
  • Priority-based (lower number = higher priority)
  • Default: deny all inbound, allow all outbound

Service Endpoints vs Private Endpoints

Service EndpointPrivate Endpoint
Traffic pathAzure backbone (still public IP)Private IP in your VNET
ExposureStill accessible from internetFully private, no internet exposure
CostFreePer hour + data cost

Azure Bastion

01:14:19
Secure Remote Access

PaaS service providing secure RDP/SSH access to VMs directly through the Azure portal — without exposing public IP addresses on VMs.

Key Benefits

  • No need to open RDP (3389) or SSH (22) ports to the internet
  • Access via HTTPS (port 443) through browser
  • Protects against port scanning and zero-day exploits on RDP/SSH
  • Deployed per VNET
  • Fully managed, no patching required
⚡
Exam Tip: Azure Bastion eliminates the need for a "jump server" / bastion host VM that you'd otherwise have to manage and patch yourself.

Azure Key Vault

01:16:18
Secrets Management

Centralised, secure storage for secrets, keys, and certificates. Eliminates the need to store credentials in code or config files.

Three Categories of Stored Items

  • Secrets — Connection strings, passwords, API keys
  • Keys — Cryptographic keys (RSA, EC) for encryption operations
  • Certificates — TLS/SSL certificates with auto-renewal
Standard Tier

Software-protected keys. Keys encrypted at rest by FIPS 140-2 Level 1.

Premium Tier

HSM-protected keys (Hardware Security Module). FIPS 140-2 Level 2/3. For regulatory requirements.

Microsoft Purview, Priva & Compliance Manager

01:33:55 – 01:52:02

Microsoft Purview Overview

Unified data governance and compliance platform. Covers data security, compliance, and data governance across on-prem, multi-cloud, and SaaS.

🏷️ Sensitivity Labels

  • Classify and protect data (e.g. Confidential, Internal)
  • Applied manually or automatically
  • Can encrypt files, add watermarks, restrict access
  • Persistent — label travels with the file

📦 Retention Labels & Policies

  • Define how long to keep or delete data
  • Retention policies = org-wide scope
  • Retention labels = item-level, more granular
  • Supports regulatory requirements (7-year retention etc.)

Data Loss Prevention (DLP)

  • Detect and prevent sharing of sensitive information (credit card numbers, NI numbers, medical data)
  • Policies applied to: Exchange, SharePoint, Teams, Endpoint devices, Power BI
  • Actions: Block, Audit, Notify, Restrict
  • Uses Sensitive Information Types (SITs) — built-in or custom patterns

Insider Risk Management

  • Detect and investigate risky activities by internal users
  • Scenarios: Data theft by departing employees, data leaks, security policy violations
  • Privacy by design — users anonymised until escalation
  • Requires E5 compliance or add-on

eDiscovery

  • Legal process of identifying, collecting electronic evidence
  • Content Search — Search across M365 (no case required)
  • eDiscovery Standard — Cases, holds, export
  • eDiscovery Premium — Advanced analytics, custodian management, review sets
  • Supports litigation holds — preserves content even if user tries to delete

Compliance Manager

  • Dashboard showing compliance posture across regulations
  • Compliance Score — Percentage of controls implemented
  • Provides improvement actions to increase score
  • Pre-built assessments for GDPR, ISO 27001, NIST, SOC 2, HIPAA etc.
  • Controls map to both Microsoft-managed and customer-managed responsibilities

Microsoft Priva

  • Privacy management solution for Microsoft 365
  • Privacy Risk Management — Identify privacy risks (data overexposure, data transfers, data minimisation)
  • Subject Rights Requests — Automate GDPR/CCPA data subject access requests

Purview Audit

  • Audit Standard — 90-day log retention of M365 audit events (included in E3)
  • Audit Premium — 1-year (or 10-year add-on) retention, high-value event logs (mail access, search queries)
  • Logs are immutable — cannot be altered
  • Used for forensic investigations and compliance reporting
⚡
Exam Tip: Know the differences: Sensitivity Labels = protect and classify data. Retention Labels = keep or delete data. DLP = prevent data from leaving. Insider Risk = detect risky user behaviour. eDiscovery = legal holds and evidence.
